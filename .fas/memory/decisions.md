[2026-06-11] **SendInstruction delivery modes: removed `retry(3)`/`guaranteed`, `mode` now optional**: Human-approved decision (discussed before applying). The two modes were declared in the `SendInstruction.mode` union but never enforced — plan-interpreter read `mode` only for logging and every send took the same single-enqueue path — so the API promised delivery guarantees the runtime does not provide. Removed rather than implemented because reliability belongs at the protocol level (ask + timeout today; the lattice activation ack/re-emit protocol per spike direct-1781143982247), and because honest implementations require machinery that does not exist yet (receiver dedup for retry; durable outbox for guaranteed). `mode` is now optional (`mode?: 'fireAndForget'`) since it has a single value; it was kept (not deleted) so richer modes can return behind real semantics. The `isSendInstruction` guard discriminates on `to`+`tell` and never depended on `mode`. Not lattice-motivated per se — the spike surfaced it; the API-honesty problem stands alone. <!-- fas:meta v=1 cf=2026-06-11T00:00:00.000Z -->
