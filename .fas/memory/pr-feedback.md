[2026-06-10] **CLI lifecycle patterns CodeRabbit catches (PR #17)** — three reusable rules for any command-surface code: (1) *Paired execution paths must share one teardown implementation* — the interactive REPL exit skipped the `flush()` that the scripted `--exec` path did; extract shutdown/drain into one function both paths call instead of duplicating steps. (2) *Never split a command string on a bare delimiter when payloads can contain it* — `script.split(';')` broke JSON like `{"text":"a;b"}`; use a quote-aware splitter (track quote state + backslash escapes) for any "commands with JSON/args" surface. (3) *Fire-and-forget promises in process-lifecycle callbacks (`rl.on('close')`, signal handlers) need `.catch` that logs and exits non-zero* — `.then`-only leaves teardown failures as unhandled rejections. **Identity-keyed registries break under normalized copies; teardown owns terminal events (PR #26, CodeRabbit)** — three reusable rules from the supervision fix round: (1) *When any registry keys on object identity (WeakMap on a behavior/config object), every later consumer — especially restart/respawn paths — must use the ORIGINAL reference, not a normalized/cloned copy.* The runtime stored `normalizeBehavior(...)` and respawned from it, silently degrading machine actors to context actors on restart; fix stores the original per path with lifecycle mirroring the normalized map. (2) *Validate every field of a new options object at the boundary, not just the discriminant* — strategy was validated at spawn but maxRestarts/withinMs were consumed raw (negative/NaN could disable bounds). (3) *Teardown owns the terminal event: exactly one `actorStopped` per stop, reason flows through the teardown function's parameter, callers never emit their own* — and any later "add a reason-coded event in path X" follow-up must check whether teardown already emitted (in the respawn-throw path it has, so the fix is a DISTINCT event type, not a second terminal event).
